In **MySQL schema design**, an index is created to make reads faster: filtering (`WHERE`), joining (`JOIN`), sorting (`ORDER BY`), grouping (`GROUP BY`), and enforcing uniqueness.

Indexes can be created in several ways.

---

## 1. Create index inside `CREATE TABLE`

This is common when defining the schema from the beginning.

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    email VARCHAR(255) NOT NULL,
    username VARCHAR(100),
    created_at DATETIME,

    INDEX idx_users_created_at (created_at),
    UNIQUE INDEX idx_users_email (email)
);
```


### Why?

Use this when the index is part of the table design and should always exist with the table.

---

## 2. Create index after table creation with `CREATE INDEX`

```sql
CREATE INDEX idx_users_username
ON users (username);
```


This is equivalent to:

```sql
ALTER TABLE users
ADD INDEX idx_users_username (username);
```


### Why?

Use this when the table already exists and you want to add an index later.

---

## 3. Create index using `ALTER TABLE`

```sql
ALTER TABLE users
ADD INDEX idx_users_created_at (created_at);
```


### Why?

Useful in migrations, because schema changes are usually written as `ALTER TABLE`.

---

# Common index types in MySQL

## 1. Primary key index

```sql
CREATE TABLE users (
    id BIGINT NOT NULL AUTO_INCREMENT,
    name VARCHAR(100),

    PRIMARY KEY (id)
);
```


### Why?

A `PRIMARY KEY` uniquely identifies each row.

In InnoDB, the primary key is also the **clustered index**, meaning the table data is physically organized around it.

Good primary keys are usually:

```sql
id BIGINT AUTO_INCREMENT PRIMARY KEY
```


or sometimes:

```sql
uuid CHAR(36) PRIMARY KEY
```


### Use when

Every table should usually have a primary key.

---

## 2. Unique index

```sql
CREATE TABLE users (
    id BIGINT PRIMARY KEY,
    email VARCHAR(255) NOT NULL,

    UNIQUE INDEX idx_users_email (email)
);
```


Or:

```sql
ALTER TABLE users
ADD UNIQUE INDEX idx_users_email (email);
```


### Why?

A unique index prevents duplicate values.

For example, if every user email must be unique, use a unique index.

```sql
INSERT INTO users (id, email)
VALUES (1, 'a@example.com');

INSERT INTO users (id, email)
VALUES (2, 'a@example.com');
```


The second insert will fail.

### Use for

- Email
- Username
- Phone number
- National ID
- Slug
- External payment/order reference

---

## 3. Normal / non-unique index

```sql
CREATE INDEX idx_orders_user_id
ON orders (user_id);
```


### Why?

Speeds up lookups where many rows may share the same value.

Example:

```sql
SELECT *
FROM orders
WHERE user_id = 10;
```


### Use for

- Foreign keys
- Status columns, sometimes
- Date columns
- Common filter columns

---

## 4. Composite index / multi-column index

```sql
CREATE INDEX idx_orders_user_id_status
ON orders (user_id, status);
```


### Why?

Useful when queries filter by multiple columns.

Example:

```sql
SELECT *
FROM orders
WHERE user_id = 10
  AND status = 'paid';
```


This index can help because both columns are in the same index.

### Important rule: leftmost prefix

For this index:

```sql
INDEX idx_orders_user_id_status_created_at (user_id, status, created_at)
```


MySQL can use it efficiently for:

```sql
WHERE user_id = 10
```


```sql
WHERE user_id = 10 AND status = 'paid'
```


```sql
WHERE user_id = 10 AND status = 'paid' AND created_at > '2026-01-01'
```


But not efficiently for:

```sql
WHERE status = 'paid'
```


because `status` is not the first column in the index.

### Use when

Your queries commonly use multiple columns together.

---

## 5. Index for foreign keys

```sql
CREATE TABLE orders (
    id BIGINT PRIMARY KEY,
    user_id BIGINT NOT NULL,

    INDEX idx_orders_user_id (user_id),

    CONSTRAINT fk_orders_user_id
        FOREIGN KEY (user_id) REFERENCES users(id)
);
```


### Why?

Foreign key columns should be indexed because they are often used in joins.

Example:

```sql
SELECT *
FROM orders o
JOIN users u ON u.id = o.user_id
WHERE o.user_id = 10;
```


MySQL usually requires an index for foreign key constraints.

---

## 6. Full-text index

```sql
CREATE TABLE articles (
    id BIGINT PRIMARY KEY,
    title VARCHAR(255),
    body TEXT,

    FULLTEXT INDEX idx_articles_title_body (title, body)
);
```


Query:

```sql
SELECT *
FROM articles
WHERE MATCH(title, body) AGAINST('mysql indexing');
```


### Why?

Use for text search.

Better than:

```sql
WHERE body LIKE '%mysql%'
```


because a normal B-tree index usually cannot help with leading wildcard searches like `'%mysql%'`.

### Use for

- Article search
- Product search
- Blog content
- Descriptions

---

## 7. Spatial index

```sql
CREATE TABLE places (
    id BIGINT PRIMARY KEY,
    location POINT NOT NULL SRID 4326,

    SPATIAL INDEX idx_places_location (location)
);
```


### Why?

Use for geographic/location queries.

Example:

- Find nearby places
- Search inside a polygon
- Distance-based filtering

---

## 8. Prefix index

```sql
CREATE INDEX idx_users_email_prefix
ON users (email(20));
```


### Why?

Indexes only the first N characters of a string column.

Useful for large `VARCHAR`, `TEXT`, or `BLOB` columns when full indexing is too large.

### Example

```sql
CREATE INDEX idx_logs_message_prefix
ON logs (message(100));
```


### Be careful

Prefix indexes may be less selective. If many values share the same prefix, the index is less useful.

---

## 9. Descending index

```sql
CREATE INDEX idx_orders_created_at_desc
ON orders (created_at DESC);
```


### Why?

Useful for queries like:

```sql
SELECT *
FROM orders
ORDER BY created_at DESC
LIMIT 20;
```


In MySQL 8+, descending indexes are supported.

---

## 10. Functional index / expression index

In MySQL 8+, you can index expressions.

```sql
CREATE INDEX idx_users_lower_email
ON users ((LOWER(email)));
```


### Why?

Useful when queries filter by a function result.

Example:

```sql
SELECT *
FROM users
WHERE LOWER(email) = 'test@example.com';
```


Without a functional index, MySQL may not use a normal index on `email` because the column is wrapped in a function.

---

# Practical examples

## Users table

```sql
CREATE TABLE users (
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    email VARCHAR(255) NOT NULL,
    phone VARCHAR(20),
    name VARCHAR(100),
    created_at DATETIME NOT NULL,

    PRIMARY KEY (id),
    UNIQUE INDEX idx_users_email (email),
    UNIQUE INDEX idx_users_phone (phone),
    INDEX idx_users_created_at (created_at)
);
```


### Why these indexes?

| Index | Reason |
|---|---|
| `PRIMARY KEY (id)` | Fast row lookup and identity |
| `UNIQUE email` | Prevent duplicate emails |
| `UNIQUE phone` | Prevent duplicate phone numbers |
| `created_at` | Useful for listing newest users |

---

## Orders table

```sql
CREATE TABLE orders (
    id BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
    user_id BIGINT UNSIGNED NOT NULL,
    status VARCHAR(30) NOT NULL,
    total_amount DECIMAL(12,2) NOT NULL,
    created_at DATETIME NOT NULL,

    PRIMARY KEY (id),
    INDEX idx_orders_user_id (user_id),
    INDEX idx_orders_user_status (user_id, status),
    INDEX idx_orders_created_at (created_at),

    CONSTRAINT fk_orders_user_id
        FOREIGN KEY (user_id) REFERENCES users(id)
);
```


### Why these indexes?

| Index | Reason |
|---|---|
| `idx_orders_user_id` | Fast lookup of user orders |
| `idx_orders_user_status` | Fast lookup of user orders by status |
| `idx_orders_created_at` | Fast sorting/filtering by date |
| FK index | Better join performance and foreign key support |

# How to choose indexes

Create indexes based on your actual queries.

## Good candidates

Columns used in:

```sql
WHERE
JOIN ON
ORDER BY
GROUP BY
UNIQUE constraints
FOREIGN KEY constraints
```


Example:

```sql
SELECT *
FROM orders
WHERE user_id = 10
ORDER BY created_at DESC;
```


Good index:

```sql
CREATE INDEX idx_orders_user_created_at
ON orders (user_id, created_at DESC);
```


---

# Things to avoid

## 1. Too many indexes

Indexes make reads faster, but they make writes slower.

Every time you run:

```sql
INSERT
UPDATE
DELETE
```


MySQL may need to update indexes too.

So do not index every column.

---

## 2. Indexing low-cardinality columns alone

Example:

```sql
status ENUM('pending', 'paid', 'failed')
```


This may not be very useful alone:

```sql
CREATE INDEX idx_orders_status ON orders(status);
```


Because many rows may have the same status.

Better:

```sql
CREATE INDEX idx_orders_status_created_at
ON orders (status, created_at);
```


or:

```sql
CREATE INDEX idx_orders_user_status
ON orders (user_id, status);
```


depending on your queries.

---

## 3. Duplicate indexes

These are often redundant:

```sql
INDEX idx_user_id (user_id),
INDEX idx_user_id_status (user_id, status)
```


The second index can often also serve queries filtering only by `user_id`.

So you may not need both, unless there is a specific reason.

---

## 4. Functions on indexed columns

This may prevent index usage:

```sql
WHERE DATE(created_at) = '2026-05-26'
```


Better:

```sql
WHERE created_at >= '2026-05-26 00:00:00'
  AND created_at <  '2026-05-27 00:00:00'
```


Then this index can be used:

```sql
CREATE INDEX idx_orders_created_at
ON orders (created_at);
```


---

# How to check if MySQL uses your index

Use:

```sql
EXPLAIN
SELECT *
FROM orders
WHERE user_id = 10
ORDER BY created_at DESC;
```


MySQL will show whether it uses an index.

---

# Simple rule of thumb

Use indexes for:

```sql
PRIMARY KEY
UNIQUE values
FOREIGN KEY columns
frequent WHERE filters
frequent JOIN columns
frequent ORDER BY / GROUP BY columns
```


But avoid adding indexes blindly, because each index costs:

- Disk space
- Slower inserts
- Slower updates
- Slower deletes
- More maintenance overhead

In short: **indexes are schema-level tools for making common reads fast and enforcing data rules like uniqueness.**

In MySQL, `KEY` is just another syntax for creating an **index**.

So this:

```sql
KEY idx_payments_user (user_id)
```


means:

```sql
CREATE AN INDEX named idx_payments_user ON column user_id
```


Inside a `CREATE TABLE`, these are basically equivalent:

```sql
KEY idx_payments_user (user_id)
```


```sql
INDEX idx_payments_user (user_id)
```


So you could write:

```sql
CREATE TABLE payments (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT UNSIGNED NOT NULL,

    KEY idx_payments_user (user_id)
);
```


or:

```sql
CREATE TABLE payments (
    id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id BIGINT UNSIGNED NOT NULL,

    INDEX idx_payments_user (user_id)
);
```


Both create a normal non-unique index.

---

## General syntax

```sql
KEY index_name (column_name)
```


or:

```sql
INDEX index_name (column_name)
```


Example:

```sql
KEY idx_payments_user (user_id)
```


Means:

| Part | Meaning |
|---|---|
| `KEY` | Create an index |
| `idx_payments_user` | Name of the index |
| `(user_id)` | Column included in the index |

---

## Composite `KEY`

This:

```sql
KEY idx_payments_payable (payable_type, payable_id)
```


means create one index using two columns together:

```sql
payable_type
payable_id
```


It helps queries like:

```sql
SELECT *
FROM payments
WHERE payable_type = 'Order'
  AND payable_id = 123;
```


This is called a **composite index** or **multi-column index**.

---

## Is `KEY` the same as foreign key?

Not exactly.

This is an index:

```sql
KEY idx_payments_user (user_id)
```


This is a foreign key constraint:

```sql
CONSTRAINT fk_payments_gateway
FOREIGN KEY (gateway_id) REFERENCES gateways(id)
```


They are different things.

### `KEY`

Creates an index for faster lookup:

```sql
KEY idx_payments_user (user_id)
```


### `FOREIGN KEY`

Creates a relationship rule between tables:

```sql
FOREIGN KEY (gateway_id) REFERENCES gateways(id)
```


A foreign key says:

> Every `payments.gateway_id` must exist in `gateways.id`.

---

## Why is it called `KEY`?

Historically, SQL databases often use the word **key** for things used to find or identify rows.

In MySQL:

```sql
KEY
```


is a synonym for:

```sql
INDEX
```


Except in special phrases like:

```sql
PRIMARY KEY
FOREIGN KEY
UNIQUE KEY
```


Those have extra meaning.

---

## Different uses of `KEY`

### 1. Normal index

```sql
KEY idx_payments_user (user_id)
```


Same as:

```sql
INDEX idx_payments_user (user_id)
```


It speeds up searches but does not enforce uniqueness.

---

### 2. Unique key

```sql
UNIQUE KEY idx_users_email (email)
```


Same as:

```sql
UNIQUE INDEX idx_users_email (email)
```


It creates an index and prevents duplicate values.

Example:

```sql
email VARCHAR(255) NOT NULL,
UNIQUE KEY idx_users_email (email)
```


Now two users cannot have the same email.

---

### 3. Primary key

```sql
PRIMARY KEY (id)
```


This uniquely identifies each row.

Only one primary key is allowed per table.

---

### 4. Foreign key

```sql
FOREIGN KEY (gateway_id) REFERENCES gateways(id)
```


This is not just an index. It enforces a relationship between tables.

---

## In your table

These lines:

```sql
KEY idx_payments_user        (user_id),
KEY idx_payments_payable     (payable_type, payable_id),
KEY idx_payments_status_time (status, created_at),
```


mean:

```plain text
Create normal non-unique indexes.
```


Equivalent version:

```sql
INDEX idx_payments_user        (user_id),
INDEX idx_payments_payable     (payable_type, payable_id),
INDEX idx_payments_status_time (status, created_at),
```


So your schema could also be written like this:

```sql
CREATE TABLE payments (
    id            BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY,
    user_id       BIGINT UNSIGNED NOT NULL,
    gateway_id    INT UNSIGNED    NOT NULL,
    payable_type  VARCHAR(32)     NOT NULL,
    payable_id    BIGINT UNSIGNED NOT NULL,
    total_amount  BIGINT          NOT NULL,
    paid_amount   BIGINT          NOT NULL DEFAULT 0,
    currency      CHAR(3)         NOT NULL DEFAULT 'IRR',
    status        VARCHAR(16)     NOT NULL,
    description   VARCHAR(255)    NULL,
    created_at    DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at    DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    paid_at       DATETIME        NULL,

    INDEX idx_payments_user        (user_id),
    INDEX idx_payments_payable     (payable_type, payable_id),
    INDEX idx_payments_status_time (status, created_at),

    CONSTRAINT fk_payments_gateway
        FOREIGN KEY (gateway_id) REFERENCES gateways(id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```


Same meaning.

## Simple answer

```sql
KEY idx_payments_user (user_id)
```


means:

> Create a normal index named `idx_payments_user` on the `user_id` column.

And in MySQL:

```sql
KEY
```


is basically the same as:

```sql
INDEX
```


when used like this inside `CREATE TABLE`.

Sample:
This `payments` table uses **InnoDB B-tree indexes**. In MySQL/InnoDB, normal `KEY` / `INDEX`, `PRIMARY KEY`, and most foreign-key-supporting indexes are implemented as **B-tree indexes**.

Your table has these indexes:

```sql
PRIMARY KEY (id)
KEY idx_payments_user        (user_id)
KEY idx_payments_payable     (payable_type, payable_id)
KEY idx_payments_status_time (status, created_at)
```


And also a foreign key:

```sql
CONSTRAINT fk_payments_gateway
FOREIGN KEY (gateway_id) REFERENCES gateways(id)
```


---

## 1. `PRIMARY KEY (id)`

```sql
id BIGINT UNSIGNED AUTO_INCREMENT PRIMARY KEY
```


This creates a **primary key index** on `id`.

Because the table uses **InnoDB**, this is also the **clustered index**.

### Why?

It gives every payment a unique identity.

Example query:

```sql
SELECT *
FROM payments
WHERE id = 100;
```


This will be very fast because `id` is the primary key.

### Important InnoDB detail

In InnoDB, the primary key stores the actual row data. Other secondary indexes point back to the primary key.

So this table is physically organized by:

```sql
id
```


---

## 2. `idx_payments_user (user_id)`

```sql
KEY idx_payments_user (user_id)
```


This is a **normal non-unique secondary index**.

### Why?

It helps find payments for a specific user.

Example:

```sql
SELECT *
FROM payments
WHERE user_id = 42;
```


It can also help joins like:

```sql
SELECT p.*
FROM payments p
JOIN users u ON u.id = p.user_id
WHERE u.id = 42;
```


### Type

```plain text
B-tree secondary index
```


### Unique?

No. One user can have many payments.

---

## 3. `idx_payments_payable (payable_type, payable_id)`

```sql
KEY idx_payments_payable (payable_type, payable_id)
```


This is a **composite secondary index**.

It indexes two columns together:

```sql
payable_type
payable_id
```


This looks like a **polymorphic association index**.

Your comments say:

```sql
-- 'Donate' | 'Order' | 'WalletCharge'
```


So a payment can belong to different payable things, for example:

```plain text
Donate       #15
Order        #15
WalletCharge #15
```


The pair is needed because `payable_id = 15` alone is not enough to know which table/entity it belongs to.

### Why?

It helps queries like:

```sql
SELECT *
FROM payments
WHERE payable_type = 'Order'
  AND payable_id = 123;
```


### Type

```plain text
Composite B-tree secondary index
```


### Leftmost-prefix rule

Because the index order is:

```sql
(payable_type, payable_id)
```


It is good for:

```sql
WHERE payable_type = 'Order'
```


and:

```sql
WHERE payable_type = 'Order'
  AND payable_id = 123
```


But it is **not ideal** for:

```sql
WHERE payable_id = 123
```


because `payable_id` is the second column, not the first.

---

## 4. `idx_payments_status_time (status, created_at)`

```sql
KEY idx_payments_status_time (status, created_at)
```


This is another **composite secondary index**.

It indexes:

```sql
status
created_at
```


### Why?

It helps queries that filter by payment status and time.

Example:

```sql
SELECT *
FROM payments
WHERE status = 'Pending'
ORDER BY created_at DESC;
```


Or:

```sql
SELECT *
FROM payments
WHERE status = 'Success'
  AND created_at >= '2026-05-01'
  AND created_at <  '2026-06-01';
```


### Type

```plain text
Composite B-tree secondary index
```


### Good for

```sql
WHERE status = 'Success'
```


```sql
WHERE status = 'Success'
  AND created_at >= '2026-05-01'
```


```sql
WHERE status = 'Pending'
ORDER BY created_at
```


### Less useful for

```sql
WHERE created_at >= '2026-05-01'
```


Because `created_at` is not the first column in the index.

If you often query only by creation time, you may also need:

```sql
KEY idx_payments_created_at (created_at)
```


---

## 5. Foreign key on `gateway_id`

```sql
CONSTRAINT fk_payments_gateway
FOREIGN KEY (gateway_id) REFERENCES gateways(id)
```


This creates a relationship from:

```sql
payments.gateway_id
```


to:

```sql
gateways.id
```


### Important MySQL/InnoDB behavior

For a foreign key, InnoDB requires an index on the child column:

```sql
gateway_id
```


If one does not exist, MySQL may automatically create one.

So this foreign key likely results in an index on:

```sql
gateway_id
```


But it is better to define it explicitly:

```sql
KEY idx_payments_gateway (gateway_id)
```


Then the foreign key is clear and the index name is controlled by you.

### Why?

It helps joins like:

```sql
SELECT p.*, g.*
FROM payments p
JOIN gateways g ON g.id = p.gateway_id;
```


And it helps InnoDB enforce referential integrity.

# Summary table

| Index | Kind | Columns | Purpose |
|---|---|---|---|
| `PRIMARY KEY` | Clustered primary index | `id` | Unique payment identity, fast lookup by id |
| `idx_payments_user` | Secondary B-tree index | `user_id` | Find payments by user |
| `idx_payments_payable` | Composite secondary B-tree index | `payable_type, payable_id` | Find payment by related entity like Order/Donate/WalletCharge |
| `idx_payments_status_time` | Composite secondary B-tree index | `status, created_at` | Filter payments by status and time |
| FK index on `gateway_id` | Secondary B-tree index | `gateway_id` | Support foreign key and gateway joins |

# In short

This table uses:

```plain text
1. Clustered primary B-tree index on id
2. Secondary B-tree index on user_id
3. Composite secondary B-tree index on payable_type + payable_id
4. Composite secondary B-tree index on status + created_at
5. Foreign-key-supporting B-tree index on gateway_id
```


The design is mainly optimized for these access patterns:

```sql
-- get payment by id
WHERE id = ?

-- get all payments of a user
WHERE user_id = ?

-- get payment for a specific payable entity
WHERE payable_type = ? AND payable_id = ?

-- get payments by status and creation time
WHERE status = ? AND created_at ...
```


One improvement I would suggest is explicitly adding:

```sql
KEY idx_payments_gateway (gateway_id)
```


before the foreign key constraint, so the schema clearly shows the index used for the foreign key.
